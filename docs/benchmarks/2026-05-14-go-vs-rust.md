# Go vs Rust Mobile Backend Benchmark - 2026-05-14

## Scope

- Server: `fedora`
- Method: side-by-side local loopback benchmark on the same server
- Production service: left untouched on `127.0.0.1:8081`
- Rust sidecar: `127.0.0.1:18081`
- Go sidecar: `127.0.0.1:18082`
- State safety: both sidecars used copied `data/` directories under `/tmp`
- Workload: read-only mobile endpoints plus login/session creation against copied state
- Tool: ApacheBench (`ab`)
- Remote raw result root: `/tmp/accord_mobile_bench_20260514_090700`

No backend source code was changed during this benchmark.

Disk I/O stayed effectively zero for the service processes during the sampled

## Throttling Observation

- No `429` responses were observed.
- No ApacheBench failed requests were observed.
- No endpoint rejected overload with an explicit app-level throttle.
- Under concurrent DB-backed reads, latency increased instead of being actively
  throttled. The practical limiter in this run was ERP/MariaDB/service capacity,
  not an explicit request rate policy.

## Compatibility Notes

The differences below did not block the mobile app in real testing. They are
documented for strict Go-contract parity review:

- `GET /v1/mobile/admin/settings`: Go returned `werka_phone` as `888862440`;
  Rust returned `+99888862440`.
- `GET /v1/mobile/admin/item-groups`: both services returned the same visible
  group names, but ordering differed.
- `GET /v1/mobile/werka/home` and `/v1/mobile/werka/pending`: Go serialized
  whole numeric quantities as `1`, while Rust serialized them as `1.0`.

## Conclusion

Rust passed the read-only smoke and load tests with zero failures. In this run,
Rust was faster on every tested hot read endpoint except supplier list, where
both services were effectively equal. The largest Rust advantage appeared on
admin item and item group reads.

Heavy stress conclusion:

- Rust remained faster on the tested DB-backed read endpoints.
- The practical overload mode was latency growth, not explicit `429`
  throttling.
- Login/session creation is the clearest overhead candidate because the copied
  `mobile_sessions.json` store is large and every login creates persistent
  session state.
- The mixed duel was heavy enough to interrupt SSH responsiveness on the host,
  so future destructive stress should use `nohup` plus fixed time budgets from
  the start.
